## 完美运行的查询示例（按复杂度排）

### 例 1 ⭐ 单年度边界变化（最稳，推荐首跑）

```powershell
python main.py "2023年有哪些点进入了城市边界"
```

**为什么能完美运行**：
- 路由命中 `["进入", "哪些点", "边界"]` → 仅 SpatialEventAgent
- gis_graph 里有 `2023年进入边界事件`，`INVOLVES_POINT` 边连到 3 个 Point

**预期回答骨架**：
```
2023 年共有 3 个点进入了边界：
- 国际会议中心 [空间分析-E1]
- 居住组团W   [空间分析-E2]
- 居住组团R   [空间分析-E3]
（2023年城市边界 ON_BOUNDARY 该事件 [空间分析-E?]）
```

**关注日志**：
- `[Master] 启动 1 个 SubAgent (进程池并发, ...)`
- `✓ SpatialEventAgent 完成 (Xs, status=done)`
- `[Master] L4 audit: N/N 引用全部合法 ✓`

## 稳定性测试

### 例 S1：诚实降级（数据缺失场景）

```powershell
python main.py "2099年火星殖民地有哪些点位"
```

**预期**：
- 2 个 SubAgent 都 `status=degraded`（answer 简短且含"未检索到"）
- Master 跳过 LLM 汇总，直接走 `_all_failed_fallback`
- 输出含"⚠ 所有分析维度均未能产出有效结果"+ 三条排查建议
- **整轮不消耗汇总 LLM token**

### 例 S2：API 不可用（鲁棒性）

```powershell
$env:DEEPSEEK_API_KEY = "invalid_key"
python main.py "2023年有哪些点进入边界"
$env:DEEPSEEK_API_KEY = "你的真实key"  # 测完恢复
```

**预期**：
- SubAgent 报 `status=failed, 尝试2次`（重试发生）
- 进程不崩，走 fallback 兜底输出

### 例 S3：路由验证

```powershell
# 纯空间问题——只调 SpatialEventAgent
python main.py "几个点进入了边界"

# 纯政策问题——只调 GraphReasoningAgent

# 综合问题——两个都调
```

## 失败排查清单

| 现象 | 原因 | 修复 |
|------|------|------|
| `status=degraded` 全 SubAgent | 数据缺失 / API 不通 | 检查 `data/full_graph/` 非空 + DEEPSEEK_API_KEY |
| 撞 `Lock is bound to a different event loop` | 共享 loop 没生效 | 确认 GraphManager 用的是 `_get_shared_loop()` |
| L4 audit 100% 幻觉率 | _split_answer_evidence 没识别 ID | 已修——见 sub_results.tool_calls 里的 raw output |
| `Executed maximum iterations` | ReAct 轮数不够 | `$env:SUBAGENT_MAX_REACT_ITERS = "8"` |
| 进程池启动慢 15-30s | 加载 LightRAG 正常开销 | 一次性，后续查询秒级 |
| 编号引用 `[空间分析-E20]` 但合法只到 E10 | LLM 真在编 | L4 已检出，看末尾警告框 |
| `vdb_chunks.json` JSON 解析报错 | nano_vectordb 写出裸 \r\n | 自动 vdb_repair 处理（启动期日志可见 REPAIRED）|
